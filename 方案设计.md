下面给你一份可以直接用于课程大作业开题/方案汇报的设计。建议题目定为：

# 面向 5G 核心网的知识查询与故障排查 Agent

## 1. 项目定位

本项目设计一个面向 5G 核心网学习、实验和排障场景的智能 Agent。它不是普通聊天机器人，而是结合 **核心网知识库、故障案例库、日志分析流程和工具调用能力**，帮助用户完成两类任务：

```text
1. 核心网知识查询
   例如：AMF、SMF、UPF、NRF 的作用是什么？
        UE 注册流程经过哪些网元？
        PDU Session 建立失败可能和哪些接口有关？

2. 核心网故障排查
   例如：UE 注册失败，AMF 日志报错，可能是什么原因？
        PDU Session 建立失败，SMF/UPF 应该怎么查？
        UERANSIM 能注册但不能上网，排查路径是什么？
```

free5GC 和 Open5GS 都适合作为项目背景环境。free5GC 是 Linux Foundation 项目，目标是实现 3GPP Release 15 及之后版本定义的 5G Core；Open5GS 是面向 5G Core 和 EPC 的开源实现，并标注为 Release-17 相关实现。([GitHub][1])

## 2. 总体目标

项目最终做成一个 Web 页面或命令行工具，用户输入核心网问题、日志片段或故障描述，系统输出结构化诊断结果。

输出格式建议固定为：

```text
问题类型：
涉及网元：
涉及接口：
可能原因：
依据/证据：
建议排查步骤：
参考文档：
```

例如用户输入：

```text
UERANSIM UE 注册成功，但是 PDU Session 建立失败，SMF 日志里出现 DNN not supported。
```

Agent 输出：

```text
问题类型：PDU Session 建立失败
涉及网元：UE、AMF、SMF、UDM、UPF
涉及接口：N1、N11、N10、N4、N6
可能原因：
1. subscriber 中没有配置对应 DNN
2. SMF 配置文件中未支持该 DNN
3. S-NSSAI 与 DNN 映射不一致
4. UPF 中缺少对应的数据面配置

建议排查：
1. 检查 subscriber 的 DNN/S-NSSAI 配置
2. 检查 smfcfg.yaml 中 DNN 配置
3. 检查 upfcfg.yaml 中 user-plane 路由配置
4. 查看 SMF 与 UPF 之间的 PFCP Session 是否建立成功
```

## 3. 技术路线

推荐技术栈：

| 模块       | 推荐方案                                    |
| -------- | --------------------------------------- |
| Agent 编排 | LangGraph                               |
| 知识库/RAG  | LlamaIndex 或 LangChain Retriever        |
| 向量数据库    | Chroma / FAISS                          |
| 前端界面     | Streamlit                               |
| 后端接口     | FastAPI，可选                              |
| 文档来源     | free5GC 文档、Open5GS 文档、3GPP 摘要、实验日志、故障案例 |
| 可选工具     | 日志搜索、配置文件检查、PCAP 字段解释、free5GC-MCP       |

LangGraph 适合这个项目，因为它本身定位是构建、管理和部署长时间运行、有状态 Agent 的底层编排框架，适合把“分类、检索、分析、工具调用、自检、生成答案”做成固定流程。([GitHub][2])

LlamaIndex 适合作为知识库检索层。它的 RAG 流程就是把外部数据加载、索引，然后根据用户问题检索相关上下文，再交给模型回答；它也支持把已有 RAG query engine 作为 Agent 工具，用于更复杂的决策流程。([Developer Documentation][3])

## 4. 系统架构设计

整体结构如下：

```text
用户输入
  ↓
问题分类模块
  ├── 核心网概念问题
  ├── 流程解释问题
  ├── 故障诊断问题
  ├── 日志分析问题
  └── 配置/抓包分析问题
  ↓
检索与工具调用模块
  ├── 核心网知识库检索
  ├── 故障案例库检索
  ├── 日志规则匹配
  ├── 配置文件检查，可选
  └── free5GC-MCP 调用，可选
  ↓
诊断推理模块
  ↓
答案自检模块
  ↓
结构化输出
```

可以把它设计成 5 个核心 Agent 节点：

| 节点                  | 作用                         |
| ------------------- | -------------------------- |
| Router Agent        | 判断用户问题类型                   |
| Knowledge Retriever | 检索核心网知识、流程说明、文档片段          |
| Fault Analyzer      | 根据日志/现象判断可能故障点             |
| Tool Agent          | 可选，调用日志搜索、配置检查、free5GC-MCP |
| Report Generator    | 生成结构化答案                    |

## 5. 数据库设计

知识库不建议一开始导入完整 3GPP 标准，工作量太大。课程项目建议先用“小而精”的数据集。

### 5.1 核心网知识库

内容包括：

```text
AMF：接入与移动性管理
SMF：会话管理
UPF：用户面转发
NRF：服务注册与发现
AUSF/UDM：鉴权与用户数据
PCF：策略控制
NSSF：切片选择
```

### 5.2 流程知识库

建议整理这些流程：

```text
UE Registration
PDU Session Establishment
Service Request
Deregistration
N4 Session Establishment
NRF Registration / Discovery
```

### 5.3 故障案例库

建议整理 15–20 个典型故障：

```text
1. UE 注册失败：PLMN 配置错误
2. UE 注册失败：AUSF/UDM 鉴权失败
3. AMF 无法注册到 NRF
4. SMF 无法发现 UPF
5. PDU Session 建立失败：DNN 不匹配
6. PDU Session 建立失败：S-NSSAI 不匹配
7. UPF PFCP Session 建立失败
8. UE 注册成功但不能上网
9. N3 GTP-U 隧道不通
10. N6 侧路由/NAT 配置错误
11. UERANSIM gNB 连不上 AMF
12. AMF NGAP SCTP 连接异常
13. MongoDB 用户数据缺失
14. WebConsole subscriber 配置错误
15. DNS 或 IP 地址配置错误
```

### 5.4 日志规则库

可以做成简单 JSON/YAML：

```yaml
- keyword: "DNN not supported"
  nf: "SMF"
  interface: "N11/N10/N4"
  possible_causes:
    - "subscriber 中没有配置对应 DNN"
    - "SMF 配置中未启用该 DNN"
    - "S-NSSAI 与 DNN 映射错误"
  next_steps:
    - "检查 subscriber 配置"
    - "检查 smfcfg.yaml"
    - "检查 UPF user-plane 配置"
```

这个模块是你们项目最容易体现“不是普通 RAG”的地方。

## 6. Agent 工作流程

### 场景一：核心网知识查询

用户输入：

```text
AMF 和 SMF 的区别是什么？
```

流程：

```text
Router 判断为“知识查询”
↓
Retriever 检索 AMF/SMF 相关文档
↓
Generator 生成解释
↓
Self-check 检查是否回答了两者功能、接口、流程位置
↓
输出
```

输出重点：

```text
AMF 偏接入和移动性管理；
SMF 偏 PDU Session 和用户面路径管理；
AMF 主要处理 UE 注册、接入控制、N1/N2；
SMF 主要处理会话建立、UPF 选择、N4 控制。
```

---

### 场景二：故障诊断

用户输入：

```text
UE 注册成功，但是不能上网。
```

流程：

```text
Router 判断为“故障诊断”
↓
Fault Analyzer 判断属于“用户面故障”
↓
Retriever 检索“注册成功但不能上网”的案例
↓
Tool Agent 可选检查日志/配置
↓
Report Generator 生成排查路径
```

输出重点：

```text
优先检查：
1. PDU Session 是否建立成功
2. SMF 是否成功选择 UPF
3. N4 PFCP Session 是否建立
4. N3 GTP-U 隧道是否正常
5. UPF 到 DN 的 N6 路由/NAT 是否正常
6. UE 是否拿到 IP 地址和默认路由
```

---

### 场景三：日志分析

用户输入一段日志：

```text
[SMF] DNN internet is not supported
```

流程：

```text
Router 判断为“日志分析”
↓
Log Matcher 匹配关键词 DNN not supported
↓
Retriever 检索 PDU Session/DNN/S-NSSAI 文档
↓
Fault Analyzer 生成原因
↓
输出结构化排查建议
```

## 8. 界面设计

Streamlit 页面可以分成三块。

### 左侧输入区

```text
问题类型：
[自动识别 / 知识查询 / 故障诊断 / 日志分析]

[文本框]

[日志文件 / 配置文件 / pcap，可选]
```

### 右侧输出区

```text
诊断结论
涉及网元
涉及接口
可能原因

```

### 底部证据区

```text

```

这个“证据区”很重要，因为老师能看到你们不是单纯让大模型瞎答，而是有检索和规则支撑。

## 9. 项目创新点

可以这样写：

第一，项目不是通用问答系统，而是面向 5G 核心网实验场景构建了专用知识库，覆盖 AMF、SMF、UPF、NRF、NAS、NGAP、PFCP、GTP-U 等核心对象。

第二，项目不是单纯 RAG，而是把问题分类、知识检索、故障规则匹配、日志分析和结构化排查流程结合起来，形成一个轻量级核心网排障 Agent。

第三，项目面向 free5GC/Open5GS/UERANSIM 实验环境，能够把“故障现象”映射到“涉及网元、涉及接口、可能原因、下一步检查命令”。

第四，项目可以扩展接入 free5GC-MCP，使 Agent 不仅能回答问题，还能进一步查看实验环境状态、查询 subscriber 或检查网元运行情况。

## 10. 最小可行版本

为了课程大作业稳妥完成，建议 MVP 做到这些就够：

```text
1. 支持核心网知识问答
2. 支持 UE 注册流程 / PDU Session 流程解释
3. 支持 10–15 个典型故障案例检索
4. 支持用户粘贴日志后进行故障分类
5. 输出结构化排查步骤
6. 有 Streamlit Web 界面
7. 有 20 个测试问题用于效果评估
```

暂时可以不做：

```text
1. 自动修改 free5GC 配置
2. 自动重启核心网
3. 自动分析完整 pcap
4. 完整 3GPP 标准文档解析
5. 复杂多 Agent 协作
```

这些可以放到“后续工作”。

## 11. 评估方法

可以设计一个小测试集。

### 11.1 知识问答测试

准备 10 个问题：

```text
AMF 的作用是什么？
SMF 和 UPF 的关系是什么？
NRF 的作用是什么？
PDU Session 建立流程是什么？
N1、N2、N3、N4、N6 分别是什么？
```

评价指标：

```text
回答是否正确
是否覆盖关键网元
是否引用知识库内容
是否结构清晰
```

### 11.2 故障诊断测试

准备 10 个故障：

```text
UE 注册失败
gNB 连不上 AMF
AMF 无法注册 NRF
PDU Session 建立失败
DNN not supported
S-NSSAI 不匹配
UPF PFCP 失败
UE 能注册但不能 ping 外网
N3 不通
N6 路由错误
```

评价指标：

```text
是否识别问题类型
是否定位相关网元
是否给出合理原因
是否给出可执行排查步骤
```

### 11.3 对比实验

可以和普通大模型直接回答做对比：

| 方法        | 是否有知识库 | 是否有故障规则 | 是否结构化输出 | 是否适合实验排障 |
| --------- | ------ | ------- | ------- | -------- |
| 普通 LLM    | 否      | 否       | 不稳定     | 一般       |
| 普通 RAG    | 是      | 否       | 一般      | 中等       |
| 本项目 Agent | 是      | 是       | 是       | 较好       |

## 14. 方案一句话总结

这个大作业可以定义为：

> 本项目面向 5G 核心网实验与学习场景，构建一个基于 RAG 和 Agent 工作流的核心网知识查询与故障排查系统。系统通过核心网知识库、故障案例库和日志规则库，将用户输入的问题或故障现象映射到相关网元、接口、可能原因和排查步骤，从而辅助 free5GC/Open5GS/UERANSIM 等实验环境中的核心网学习与调试。
